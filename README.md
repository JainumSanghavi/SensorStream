# SensorStream

[![CI](https://github.com/JainumSanghavi/SensorStream/actions/workflows/ci.yml/badge.svg)](https://github.com/JainumSanghavi/SensorStream/actions/workflows/ci.yml)

A real-time industrial sensor **ingestion and alerting backend**. Facilities
stream high-frequency telemetry (flow, temperature, energy, pressure); the
service bulk-ingests batches, fires alerts on threshold breaches, and serves
fast time-windowed queries and aggregations over a readings table sized in the
millions.

This is a portfolio project. It deliberately stays narrow and goes deep on the
backend patterns that matter at scale: **bulk ingestion, keyset pagination,
SQL aggregation, composite indexing, a consistent API contract, and a measured
performance proof.** Every component is meant to demonstrate a transferable
pattern rather than feature breadth.

> All data is **synthetic**, generated by `scripts/seed.py` to exercise query
> performance at realistic volume (~2–3M readings).

## Quickstart

```bash
# 1. Run the whole stack (Postgres + API) with one command.
cp .env.example .env
docker compose up --build
#   API:         http://localhost:8000        docs: http://localhost:8000/docs
#   Grafana:     http://localhost:3000        (anonymous viewer; dashboard auto-provisioned)
#   Prometheus:  http://localhost:9090
#   Jaeger:      http://localhost:16686       (traces)

# 2. (optional) Load synthetic telemetry. Run from the host against the
#    exposed Postgres, or `docker compose exec api ...`.
export DATABASE_URL=postgresql+asyncpg://sensor:sensor@localhost:5432/sensorstream
python -m scripts.seed --readings 2000000 --reset

# 3. (optional) Watch live ingestion + alerting.
python -m scripts.simulator --sensor 1 --interval 2
```

Run the tests (needs the compose Postgres up):

```bash
docker compose up -d db
export TEST_DATABASE_URL=postgresql+asyncpg://sensor:sensor@localhost:5432/sensorstream
pip install -r requirements.txt
pytest
```

### Schema

```
facilities                sensors                         readings  (high volume)
──────────                ───────                         ────────
id           PK           id              PK              id           PK  BIGINT
name                      facility_id     FK ─▶ facilities.id          sensor_id    FK ─▶ sensors.id
region                    name                            value        DOUBLE PRECISION
                          type   {flow|temperature|       recorded_at  TIMESTAMPTZ
                                  energy|pressure}
                          alert_threshold                 INDEX (sensor_id, recorded_at DESC)

alerts
──────
id            PK  BIGINT
sensor_id     FK ─▶ sensors.id        INDEX (sensor_id)
severity      {low|medium|high}
status        {active|resolved}       INDEX (status)
triggered_at  TIMESTAMPTZ
reading_value DOUBLE PRECISION
```

DDL lives in [`migrations/001_schema.sql`](migrations/001_schema.sql) — the
source of truth, applied idempotently on startup by `app/init_db.py`. The ORM in
`app/models.py` mirrors it for typed query construction; we do **not** rely on
`create_all` so the documented indexes and CHECK constraints are guaranteed.

## API

All list endpoints return an envelope `{ "data": [...], "pagination": {...} }`.
All errors return one shape: `{ "error": { "code", "message", "details" } }`.
JSON fields are **camelCase**; timestamps are **ISO 8601 UTC**.

**Auth:** the mutating endpoints (`POST` ingest, `PATCH`, `DELETE`) require an
`X-API-Key` header when `API_KEY` is set. With `API_KEY` unset, write auth is
disabled (a startup warning is logged) so local dev and tests stay frictionless.
A real deployment would issue per-client keys or JWTs — see the scale notes.

| Method & path                          | Purpose                                                        |
| -------------------------------------- | -------------------------------------------------------------- |
| `POST /sensors/{id}/readings`          | **Batch** ingest `{readings:[{value, recordedAt}]}` → `201` summary `{ingested, alertsTriggered}`. Bulk insert; fires alerts on breach. |
| `GET /sensors`                         | List sensors, OFFSET pagination (`page`, `pageSize`).          |
| `GET /sensors/{id}`                    | One sensor, `404` if missing.                                  |
| `GET /sensors/{id}/readings`           | **Keyset/cursor** pagination, newest-first. `?from= &to= &limit= &cursor=`. |
| `GET /sensors/{id}/stats?interval=`    | Time-bucketed `avg/min/max/count` (`hour`\|`day`).             |
| `GET /alerts?status=active`            | Alerts filtered by `status` (and optional `sensorId`).         |
| `PATCH /sensors/{id}`                  | Partial update (e.g. `alertThreshold`).                        |
| `DELETE /sensors/{id}`                 | `204 No Content`.                                              |
| `GET /health`                          | Liveness + DB connectivity.                                    |
| `GET /metrics`                         | **Prometheus** exposition: request rate, latency histograms, ingest/alert counters. |
| `GET /metrics/summary`                 | Human-readable JSON ops summary: total readings, active alerts, ingestion rate. |

Example — batch ingest:

```bash
curl -X POST localhost:8000/sensors/1/readings -H 'content-type: application/json' -d '{
  "readings": [
    {"value": 142.3, "recordedAt": "2026-06-19T10:00:00Z"},
    {"value": 161.7, "recordedAt": "2026-06-19T10:00:05Z"}
  ]
}'
# -> 201 {"sensorId":1,"ingested":2,"alertsTriggered":1}
```

## Observability

`docker compose up` brings up a full local monitoring stack alongside the API —
**Prometheus** (scrape), **Grafana** (dashboards), and **Jaeger** (traces) — so
the system is observable out of the box, no cloud account required.

![SensorStream Grafana dashboard](docs/grafana-dashboard.png)

- **Metrics.** `/metrics` serves Prometheus exposition format. HTTP-level
  metrics (request rate, latency histograms, in-flight requests) are
  auto-instrumented; domain metrics — `sensorstream_readings_ingested_total`,
  `sensorstream_alerts_triggered_total`, and an `sensorstream_ingest_batch_size`
  histogram — are declared in `app/metrics.py` and incremented on the ingest
  path. Prometheus scrapes it every 5s (`ops/prometheus.yml`).
- **Dashboard.** Grafana auto-provisions a datasource and the **SensorStream**
  dashboard (`ops/grafana/dashboards/sensorstream.json`): request rate, p50/p95/p99
  latency, ingestion throughput, alert rate, 5xx error rate, and ingest batch-size
  distribution. It loads on startup with no manual import.
- **Tracing.** When `OTEL_EXPORTER_OTLP_ENDPOINT` is set (it is, in compose),
  OpenTelemetry instruments FastAPI and SQLAlchemy and exports spans to Jaeger,
  so a slow request can be traced down to the individual SQL statement. Tracing
  is fully optional and defensive — unset the env var and it's a clean no-op, so
  local runs and the test suite need no collector. See `app/observability.py`.

## Design decisions & why

### 1. Batch + bulk insert on ingestion
`POST /sensors/{id}/readings` takes a **list** of readings and writes them with a
single set-based `INSERT ... VALUES (...), (...), ...` (SQLAlchemy
`session.execute(insert(Reading), rows)`), never a per-row loop. High-frequency
telemetry arrives in bursts; one multi-row statement amortizes parse/plan and
network round-trips across the whole batch. Alerts for breaching readings are
derived **in Python** from the same in-memory batch and inserted in a second
set-based statement — no re-reading the rows we just wrote. The whole batch runs
in one transaction, so a failure leaves nothing partially written. See
`app/services.py`.

### 2. Keyset (cursor) pagination on readings — not OFFSET
`OFFSET n LIMIT k` makes Postgres generate and discard the first `n` rows on
every request; page 10,000 of a multi-million-row table walks ~500k rows just to
throw them away — cost grows linearly with depth. **Keyset** pagination instead
encodes the last row seen as a cursor `(recordedAt, id)` and asks for rows
strictly *before* it:

```sql
WHERE sensor_id = $1 AND (recorded_at, id) < ($cursor_ts, $cursor_id)
ORDER BY recorded_at DESC, id DESC LIMIT $k
```

With the composite index this seeks straight to position and reads `k` rows —
**constant-time regardless of depth**. The `id` tie-breaker makes ordering total,
so no row is skipped or duplicated when two readings share a timestamp. See
`app/pagination.py`.

### 3. Time-bucketed aggregation in SQL
`GET /sensors/{id}/stats` pushes the work to the database:
`date_trunc(interval, recorded_at)` collapses each reading onto its bucket and
`GROUP BY` computes `avg/min/max/count` per bucket in a single indexed pass. The
DB aggregates millions of rows far faster than streaming them to the app would,
and the `WHERE (sensor_id, recorded_at)` rides the composite index so only the
requested slice is touched. See `app/routers/stats.py`.

### 4. Composite indexing with documented rationale
`readings (sensor_id, recorded_at DESC)` is the keystone index. Equality column
(`sensor_id`) first narrows to one sensor's slice; the range/sort column
(`recorded_at`) second lets the planner satisfy both the time filter **and** the
`ORDER BY` from the index with no separate sort. `alerts(status)` keeps the hot
"active alerts" lookup cheap as resolved alerts accumulate; `alerts(sensor_id)`
and `sensors(facility_id)` index the FK drill-downs Postgres doesn't auto-index.
Each index carries an inline rationale in `migrations/001_schema.sql`, and the
measured payoff is in **[docs/PERFORMANCE.md](docs/PERFORMANCE.md)**: on 2M rows
the readings query goes from a **27.3 ms seq-scan-plus-sort to a 0.069 ms index
scan (~395×)**.

### 5. Shared error envelope + structured logging
Custom exception handlers normalize our `AppError`, FastAPI `HTTPException`, and
validation errors (422) into one `{error:{code,message,details}}` shape — clients
parse errors the same way everywhere. A middleware emits one structured JSON log
line per request (method, path, status, latency) — greppable and aggregator-ready.

### 6. Async where it earns its keep
The app uses async SQLAlchemy + asyncpg end-to-end. Ingestion and queries are
**I/O-bound** (waiting on Postgres), exactly where async concurrency lets one
worker serve many in-flight requests without thread-per-request overhead. CPU-
bound work (alert classification over a batch) stays plain synchronous Python —
async there would add complexity for no gain.

## What I'd do at scale / what I deliberately left out

This is a focused demo. In a production deployment at real industrial volume I
would reach for:

- **Time-based partitioning of `readings`** (declarative range partitions by
  day/month). Keeps indexes small and hot, makes retention a cheap
  `DROP PARTITION` instead of a mass `DELETE`, and lets the planner prune whole
  partitions outside the queried window. The single-table design here is the
  honest starting point; partitioning is the first thing I'd add as the table
  grows.
- **A dedicated time-series store** (TimescaleDB hypertables, or ClickHouse for
  analytical rollups). Continuous aggregates / materialized rollups would serve
  the stats endpoint from pre-computed buckets instead of scanning raw rows.
- **Async ingestion via a queue** (Kafka / Kinesis / SQS) with worker consumers.
  The HTTP endpoint would validate and enqueue; workers batch-write to Postgres.
  This decouples ingest spikes from write throughput, adds backpressure and
  replay, and lets ingestion scale independently of the query API.
- **Caching aggregation results** (Redis) for popular sensor/interval combos,
  with short TTLs — stats queries are read-heavy and tolerate slight staleness.
- **Horizontal scaling**: the API is stateless, so it scales out behind a load
  balancer trivially; the database is the bottleneck, addressed by read replicas
  for query traffic, connection pooling (PgBouncer), and the partitioning above.
- **Metrics at scale**: the Prometheus + OpenTelemetry stack here is real but
  single-instance. Across many replicas, production would scrape each pod
  (service discovery), centralize with remote-write to a long-term store
  (Thanos/Mimir/managed Grafana Cloud), and sample traces rather than export
  every span.
- **Alert lifecycle**: resolve/ack endpoints, dedup/debounce so a stuck sensor
  doesn't emit thousands of alerts, and notification fan-out — left out to keep
  the alerting path focused on the ingestion-time trigger pattern.

These are discussed, not built, on purpose: the goal is to show I know *where*
the system breaks and *what* the next move is, without over-building a portfolio
project.
